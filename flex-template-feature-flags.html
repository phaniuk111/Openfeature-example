Key insight Flag files are baked into the container image. In FILE mode, all Dataflow workers read from their local /opt/flags/flags-{env}.json. To change a flag value you publish a new image — there is no hot-reload during a running job.

1 pom.xml – new dependencies & shade plugin

The Flex Template update adds two Beam dependencies and a maven-shade-plugin to produce a fat JAR the Dockerfile can copy in.

# 3. Tell the launcher which class to run

ENV FLEX_TEMPLATE_JAVA_CLASSPATH="/template/pipeline.jar"

What gets baked in

Both the fat JAR and flags/ directory are copied at image build time. Every worker that Dataflow spins up has /opt/flags/flags-dev.json, flags-staging.json, and flags-prod.json available locally.

Why local files work

flagd FILE mode reads from the local filesystem of whatever process is running. Inside a Dataflow worker container, /opt/flags/ is always present because it was baked into the image — no GCS fetch needed.

Changing flags

Since files are baked in, a flag change means: edit JSON → mvn package → docker build & push → publish new template spec → re-run the job. No hot-reload is possible.

4 EvaluateFlagsFn – how OpenFeature lives inside a DoFn

⚠️ Critical pattern: double-checked locking for the shared Client Dataflow can run many DoFn instances on the same JVM. OpenFeature's singleton API must be initialised exactly once. The code uses a volatile static field + synchronized block to guarantee this safely. 
// Shared across all DoFn instances on the same worker JVM
private static final Object LOCK = new Object();
private static volatile Client sharedClient;

@Setup                         // called once per DoFn instance (not per element)
public void setup() throws Exception {
    if (sharedClient != null) { client = sharedClient; return; }  // fast path

    synchronized (LOCK) {
        if (sharedClient == null) {                                 // double-check
            Path flagPath = Paths.get(flagFilePath).toAbsolutePath();
            // flagFilePath = /opt/flags/flags-prod.json (baked in image)

            OpenFeatureAPI api = OpenFeatureAPI.getInstance();
            api.setProviderAndWait(new FlagdProvider(
                FlagdOptions.builder()
                    .resolverType(Config.Resolver.FILE)
                    .offlineFlagSourcePath(flagPath.toString())
                    .deadline(500).build()));

            sharedClient = api.getClient("dataflow-flex-template");
        }
        client = sharedClient;
    }
}

@ProcessElement               // called for every input element
public void processElement(ProcessContext context) {
    EvaluationContext ctx = new ImmutableContext(
        Map.of("environment", new Value(appEnv)));  // from --appEnv param

    boolean newUi   = client.getBooleanValue("new-ui",             false,         ctx);
    Integer batch   = client.getIntegerValue("dataflow-batch-size", 100,           ctx);
    String  banner  = client.getStringValue( "welcome-banner",      "fallback",    ctx);

    context.output("input=" + context.element()
        + ", new-ui=" + newUi + ", batch=" + batch + ", banner=" + banner);
}

@Setup vs @ProcessElement

@Setup runs once per DoFn instance when a worker bundle starts. OpenFeature is initialised here — expensive but only paid once. @ProcessElement runs for every record; by then client is already ready.

No @Teardown shutdown

Unlike the simple local runner, there is no api.shutdown() here. The Dataflow runner manages worker lifecycle. The flagd file-watcher thread will terminate naturally when the worker JVM exits at job completion.

5 Adding a new feature flag – what to change

Only two file changes are needed, plus a rebuild and redeploy:

flags/flags-prod.json (also flags-dev.json, flags-staging.json)
{ "flags": { "new-ui": { ... }, "dataflow-batch-size":{ ... }, "welcome-banner": { ... }, "my-new-flag": { "state": "ENABLED", "variants": { "on": true, "off": false }, "defaultVariant": "off" } } }
EvaluateFlagsFn.java – inside @ProcessElement
+ boolean myFlag = client.getBooleanValue("my-new-flag", false, ctx); context.output("input=" + context.element() + + ", my-new-flag=" + myFlag + ", new-ui=" + newUi + ...);
⚠️ Don't forget to rebuild and republish the image After changing the JSON files you must: mvn -DskipTests package → docker build & push → publish new template spec. The running job won't see the change until restarted with the new image.

6 Full build & deploy flow

# 1. Build shaded JAR
mvn -DskipTests package
# → target/maven-dataflow-example-0.0.1-SNAPSHOT-all.jar

# 2. Build & push container image
docker build -f flex-template/Dockerfile \
  -t gcr.io/<PROJECT>/openfeature-flex-template:latest .
docker push gcr.io/<PROJECT>/openfeature-flex-template:latest

# 3. Build (publish) the Flex Template spec to GCS
gcloud dataflow flex-template build gs://<BUCKET>/templates/openfeature.json \
  --image gcr.io/<PROJECT>/openfeature-flex-template:latest \
  --sdk-language JAVA \
  --metadata-file flex-template/metadata.json

# 4. Run the Flex Template job
gcloud dataflow flex-template run openfeature-demo \
  --template-file-gcs-location gs://<BUCKET>/templates/openfeature.json \
  --region us-central1 \
  --parameters appEnv=prod \
  --parameters flagFilePath=/opt/flags/flags-prod.json \
  --parameters output=gs://<BUCKET>/openfeature-demo/output
