OpenFeature – Context & Evaluation Modes Explained

Spring Boot app · Maven Dataflow · flagd FILE mode

There are three distinct modes for how OpenFeature evaluates flags in this codebase. The difference comes down to two questions: where does the Context come from? and which JSON file holds the rules? Everything else follows from those two answers.

MODE 1

API Endpoint
(Dynamic Context)

Spring Boot · FeatureController
Where does Context come from?
📡 Caller passes it in the URL
GET /api/flags?environment=prod&userId=u123
The frontend/caller decides what's in the "sticky note".
Which file has the rules?
Whichever file application.yml points to — controlled by FLAG_MODE env var:
per-env → flags-{APP_ENV}.json
targeting → flags-targeting.json
How it works: The controller receives ?environment=prod&userId=u42, passes both to FeatureEvaluationService.evaluate(), which builds an ImmutableContext and hands it to flagd. flagd runs the rules against that context.
Use when: A React / Angular / mobile app needs to ask the backend "what flags are enabled for this specific user right now?" The caller provides the identity.
MODE 2

Simple Per-Env
Files

Spring Boot · flags-dev/staging/prod.json
Where does Context come from?
🖥️ Backend's own APP_ENV
Set at deploy time: APP_ENV=prod.
No per-user data. Same answer for everyone.
Which file has the rules?
flags-dev.json flags-staging.json flags-prod.json
One file per environment. No targeting block — just a hardcoded defaultVariant.
How it works: flagd completely ignores the Context. It opens the file for the current environment and returns whatever boolean/integer/string is hardcoded as defaultVariant. Context is still sent but has no effect.
Use when: You want to turn a feature entirely ON or OFF for all users in an environment. Simple on/off per deploy — no conditional logic needed.
MODE 3

One Smart File
(Targeting Rules)

Spring Boot · flags-targeting.json
Where does Context come from?
🧠 Backend's own env + any user data
environment=prod + optional userId.
The backend knows its own identity; caller can add more.
Which file has the rules?
flags-targeting.json
One master file with "targeting": { "if": [...] } JsonLogic rules for all envs and users.
How it works: flagd reads the if/then JsonLogic in the file, compares it to the Context "sticky note", and makes a dynamic decision. e.g. "if environment == prod → return 'on'".
Use when: You have complex rules — e.g. "ON for prod + staging, OFF for dev" or "ON for userId starting with 'beta-'" — and want all logic in one file.
How each mode looks in actual code

Mode 2 – Per-Env Files

Dataflow Flex Template

The caller drives everything. Context is built from the HTTP request parameters.

// FeatureController.java
@GetMapping
public FeatureResult getFlags(
    @RequestParam(required = false) String environment,  // from ?environment=prod
    @RequestParam(required = false) String userId) {     // from ?userId=u42
    return featureEvaluationService.evaluate(environment, userId);
}

// FeatureEvaluationService.java — builds the "sticky note"
private EvaluationContext buildContext(String environment, String userId) {
    Map<String, Value> map = new HashMap<>();
    map.put("environment", new Value(environment));   // e.g. "prod"
    if (hasText(userId))
        map.put("userId", new Value(userId));           // e.g. "u42"
    return new ImmutableContext(map);
}

// Then evaluated against whichever file FLAG_MODE points to:
boolean newUi = client.getBooleanValue("new-ui", false, context);
The rules file is set at app startup via FLAG_MODE=targeting or FLAG_MODE=per-env. The caller can't change which file is used — only the Context values they send.

The JSON file has no targeting block — just a hardcoded defaultVariant. Context is ignored.

// flags-prod.json — no "targeting" block at all
{
  "flags": {
    "new-ui": {
      "state": "ENABLED",
      "variants": { "on": true, "off": false },
      "defaultVariant": "off"    ← flagd returns this regardless of context
    },
    "dataflow-batch-size": {
      "state": "ENABLED",
      "variants": { "small": 100, "medium": 500, "large": 2000 },
      "defaultVariant": "large"  ← always 2000 in prod
    }
  }
}

// OpenFeatureConfig.java — file chosen at startup
fileName = "flags-" + appEnv + ".json";  // e.g. flags-prod.json
// The context you pass in processElement / evaluate() has zero effect here
This is the simplest and most predictable mode. If you need "always ON in dev, always OFF in prod" — use this. No logic, no surprises.

One file, JsonLogic rules, context is actively evaluated against the rules at runtime.

// flags-targeting.json — "targeting" block with if/or JsonLogic
{
  "new-ui": {
    "state": "ENABLED",
    "variants": { "on": true, "off": false },
    "defaultVariant": "off",                 ← fallback if no rule matches
    "targeting": {
      "if": [
        { "or": [
            { "==": [ { "var": "environment" }, "prod" ] },
            { "==": [ { "var": "environment" }, "production" ] }
          ]
        },
        "on",          ← prod/production → ON
        { "==": [ { "var": "environment" }, "staging" ] },
        "on",          ← staging → ON
        "off"           ← everything else (dev) → OFF
      ]
    }
  }
}

// FeatureEvaluationService sends "environment" in context:
map.put("environment", new Value(effectiveEnvironment));
// flagd reads context.environment → runs the if/or → returns "on" or "off"
The var key in JsonLogic must exactly match a key you put into the ImmutableContext. If the spelling differs, the rule silently falls through to defaultVariant.

Flex Template uses Mode 2 (per-env files) but the file is baked into the container image and passed via --flagFilePath. Context only has environment (from --appEnv).

// FlexTemplateOptions — caller sets env and file path as pipeline params
--appEnv=prod
--flagFilePath=/opt/flags/flags-prod.json   ← baked into Docker image

// EvaluateFlagsFn.@Setup — initialised once per worker JVM
OpenFeatureAPI api = OpenFeatureAPI.getInstance();
api.setProviderAndWait(new FlagdProvider(
    FlagdOptions.builder()
        .resolverType(Config.Resolver.FILE)
        .offlineFlagSourcePath(flagFilePath)   ← /opt/flags/flags-prod.json
        .deadline(500).build()));

// @ProcessElement — context built per element (⚠️ should be moved to @Setup)
EvaluationContext ctx = new ImmutableContext(
    Map.of("environment", new Value(appEnv)));  ← same value every time

boolean newUi  = client.getBooleanValue("new-ui", false, ctx);
Integer batch  = client.getIntegerValue("dataflow-batch-size", 100, ctx);
✅ Fixed in latest version: The developer addressed all reported issues. EvaluationContext is now built once in @Setup and stored as a transient evaluationContext field. The gs:// path guard, try/catch with api.shutdown() on failure, pipeline.run().waitUntilFinish(), and the removal of the risky static volatile sharedClient pattern are all in the updated code.
Which mode should I use? — Decision guide
Pick your mode in 3 questions
Q1
Does a frontend / mobile caller need flag values personalised per user or per session?
→ Yes: Mode 1 – API endpoint — caller provides the Context.
→ No: go to Q2.
Q2
Do you need conditional logic (e.g. "ON for staging + prod, OFF for dev") or user-based rules in a single file?
→ Yes: Mode 3 – flags-targeting.json with FLAG_MODE=targeting.
→ No: go to Q3.
Q3
Do you just want a feature hardcoded ON or OFF per environment with no conditions?
→ Mode 2 – per-env files (flags-dev/staging/prod.json). Simplest, most predictable.
DF
Running on Dataflow Flex Template? You're always in Mode 2 — the file is baked into /opt/flags/ and chosen via --flagFilePath. Targeting rules are possible only if you point to flags-targeting.json and pass the right context keys in @Setup.

🐳 Flex Template — why it only does Mode 2 by default

In the Flex Template, the Context only contains environment (set from --appEnv). There is no userId and no per-request data — Dataflow workers process records in batch, not as individual user sessions. If you pointed --flagFilePath at flags-targeting.json and put environment-based if/or rules in it, Mode 3 would work too. But user-level targeting (Mode 1) doesn't apply to batch pipelines by nature.
